# Операционные правила репозитория

Этот документ описывает локальные доступы, работу с секретами и поддержку самих инструкций репозитория.

## Область действия

- Используй этот файл, когда задаче нужны локальные токены, аутентифицированные действия или обновление самих инструкций.
- Для bootstrap и маршрутизации смотри `LOCAL_AGENT_START.md`.
- Для git/PR правил смотри `docs/agent/GIT_WORKFLOW.md`.
- Для review смотри `docs/agent/REVIEW_GUIDELINES.md`.

## Локальные файлы и доступы

- Используй `GIT_USERNAME` и `GIT_TOKEN` для git-действий и операций с удалённой платформой, если нужны аутентифицированные действия.
- Для `gh` используй `GH_TOKEN`; если токен загружается из `.env`, предпочитай `set -a && source .env && set +a`, чтобы переменные точно экспортировались в окружение команды.
- Админские доступы используй только для owner-level операций над репозиторием.
- Не полагайся на парольную аутентификацию, если этого явно не требуют.
- Проверяй `.env` только тогда, когда задаче действительно нужны локальные доступы или токены.

## Локальный рабочий каталог

- По умолчанию работай только в текущем локальном каталоге репозитория.
- Не создавай и не переключайся на альтернативные локальные папки проекта, соседние клоны репозитория или `git worktree`, если пользователь явно не попросил такой режим.
- Если на машине уже есть другие локальные копии репозитория, игнорируй их и продолжай работать в текущем каталоге, если пользователь явно не указал иное.

## Поддержка инструкций

- Не дублируй правило в нескольких документах: обновляй один профильный файл, который указан в `LOCAL_AGENT_START.md`.
- Если задача меняет правила работы репозитория, обновляй профильный документ по его зоне ответственности, а не `AGENTS.md` по инерции.
- Если в ходе задачи обнаружились стабильные `gh`-команды, API-запросы или GraphQL mutation, добавляй их в `docs/agent/GITHUB_AUTOMATION.md`.
- Если меняется реальный рабочий процесс по задачам, обновляй `docs/TRACKER_WORKFLOW.md`.
- Если меняются правила ветвления, handoff в review или проверки перед push, обновляй `docs/agent/GIT_WORKFLOW.md`.
- Если меняются статусы, labels или правила project board, обновляй `docs/agent/BOARD_WORKFLOW.md` и при необходимости `docs/BOARD.md`.
