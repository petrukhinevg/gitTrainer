# Программа приложения

## Цель документа

Этот документ фиксирует текущее состояние продукта в репозитории, недостающие блоки и практическую границу до первого демо. Он нужен как быстрый продуктовый срез без необходимости каждый раз восстанавливать картину по коду и roadmap.

## Что уже есть

### Backend

- API каталога сценариев: получение списка сценариев, фильтрация, сортировка и обработка fixture-source режимов
- API деталей сценария: task preview, инструкции, шаги, annotations и статический контекст репозитория
- API сессий: старт учебной сессии по сценарию
- API submissions: отправка ответа и получение результата проверки
- Базовая валидация ответов для `command_text` на fixture-правилах
- Тесты контроллеров и валидации для текущего MVP-среза

### Контент и модель

- Набор стартовых сценариев для MVP-каталога
- Статические repository cues: ветки, коммиты, файлы, annotations
- Структура task-content: goal, instructions, ordered steps, supporting annotations

### Системный каркас

- Spring Boot backend-проект
- Разделение backend-кода по capability packages: `scenario`, `session`, `validation`
- Документированные roadmap и workflow для развития продукта

## Чего пока не хватает

### До полноценного учебного цикла

- Объяснения ошибок глубже простого correctness outcome
- Retry loop с сохранением контекста и прогрессивными подсказками
- Durable storage прогресса и истории попыток
- Рекомендации следующего упражнения и user-facing progress summary

### До пользовательского frontend-контура

- Полноценный экран каталога, подключённый к backend API
- Полноценный экран упражнения с рабочим UX для решения сценария
- Клиентская интеграция старта сессии, отправки ответа и показа feedback
- Видимый progress flow для уже решённых и доступных следующих упражнений

### До более реалистичной проверки

- Проверка не только текста команды, но и результата действия
- Более гибкая модель answer types
- Более предметная validation policy по сценариям
- Связка проверки с состоянием реального или более правдоподобно смоделированного Git-workspace

## Что отделяет проект от первого демо

Чтобы получить первое кликабельное демо, в котором можно зайти, выбрать задание и пройти базовый сценарий, нужны как минимум такие блоки:

1. Поднять frontend shell с маршрутами каталога и упражнения.
2. Подключить frontend к уже существующим backend API сценариев и сессий.
3. Собрать экран упражнения с task context, repository cues, полем ввода ответа и feedback panel.
4. Довести transport flow до состояния: открыл сценарий -> стартовал сессию -> отправил ответ -> увидел результат.
5. Добавить минимальный пользовательский путь возврата к каталогу и повторной попытки.

## Что даст завершение текущего roadmap

Если текущий roadmap будет реализован целиком, пользователь должен получить MVP, в котором можно:

- открыть приложение
- выбрать упражнение
- увидеть контекст задачи и состояние репозитория
- отправить ответ
- получить проверку и объяснение
- видеть прогресс и рекомендации следующего шага

## Важная граница MVP

Текущий roadmap описывает управляемый учебный workspace. Это означает, что целевая версия MVP позволяет решать Git-задачи в рамках guided experience, но не обязана быть полноценной живой Git-песочницей с реальным выполнением команд в настоящем репозитории.

Если позже продукт пойдёт в сторону полноценной Git-sandbox execution model, это уже отдельный большой слой сверх текущего MVP.
