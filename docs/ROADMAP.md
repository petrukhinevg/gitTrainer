# Дорожная карта

## Цель

Продукт — это single-page web application для практики Git через короткие структурированные упражнения. Пользователь должен иметь возможность открыть приложение, выбрать сценарий вроде ветвления, ребейза, разрешения конфликтов или очистки истории и пройти управляемую задачу с немедленной проверкой.

Основная пользовательская проблема — разрыв между чтением теории по Git и уверенным использованием команд в контексте. MVP должен сосредоточиться на практической повторяемости: понятные задачи, наблюдаемое состояние репозитория, валидация ответов и объяснения, которые помогают понять ошибки.

## Правила использования roadmap

- Этот roadmap должен описывать только product-level `parent issues`.
- Не используй этот файл как прямой источник задач implementation-уровня.
- Один parent issue может включать backend-работу, frontend-работу или обе стороны сразу.
- Детальные implementation tasks должны жить в side-specific roadmap-файлах.
- Предпочитай parent issues, которые описывают один пользовательский vertical slice или один блок планирования с чёткими границами.
- Child tasks внутри одного parent issue должны оставаться независимо реализуемыми от базового состояния parent issue.
- Держи side-specific roadmaps в состоянии task-ready, чтобы все перечисленные child tasks можно было создать сразу после initial epic commit.
- Если нескольким child tasks нужна одна и та же минимальная seam, например DTO skeletons, provider interfaces, route placeholders или fixture schemas, выноси её в initial epic commit, а не в одну из child tasks.
- Предпочитай декомпозицию по стабильным seam-границам вроде shell, content или policy, а также rendering или integration, а не по строгой схеме «сначала контракт, потом всё остальное».
- Начинай с грубой декомпозиции child tasks и детализируй её позже только тогда, когда это улучшает параллельность, уменьшает блокировки или даёт более чистые review units.
- Если live-hookup или sequencing step потребует код из незавершённой sibling task, перенеси этот шаг в отдельную follow-up task или в другой parent issue вместо цепочки зависимых sibling tasks.

## Определение MVP

MVP завершён, когда:

1. пользователь может открыть приложение и выбрать сценарий из небольшого каталога Git-упражнений
2. пользователь может выполнить упражнение, вводя или выбирая Git-действия в управляемом workspace
3. система валидирует попытку и объясняет, почему ответ правильный или неправильный
4. приложение сохраняет прогресс, чтобы пользователь видел завершённые упражнения и мог возвращаться за новыми

## Как использовать в трекере

- Создавай parent issues из этого файла, когда работа описывает один значимый продуктовый блок.
- Если parent issue затрагивает несколько технических сторон, декомпозируй его в side-specific tasks в соответствующих roadmap-файлах.
- Если parent issue влияет только на одну сторону, держи детальную разбивку только в roadmap этой стороны.
- Перед созданием child tasks убедись, что initial epic commit несёт только общие baseline seams, которые нужны всем child tasks для независимого ветвления.
- Когда backend и frontend описывают один и тот же пользовательский slice, но остаются независимо реализуемыми, держи их под одним parent issue и связывай через project field `Pairs with`.
- Предпочитай side-specific task sets, которые можно связывать по функциональному slice, чтобы epic можно было закрыть только при взаимной готовности backend и frontend по одному и тому же поведению.
- Независимые backend-only или frontend-only tasks также допустимы, если они дают завершённый результат без зависимости от незавершённых sibling tasks.
- Не форсируй one-to-one pairing там, где одна сторона имеет легитимную самостоятельную работу, например content modeling, policy definition, state handling или presentation-only improvements.
- Держи текущую очередь и сопоставление статусов синхронизированными с `docs/BOARD.md`.

## Родительские задачи

### Родительская задача 1. MVP просмотра каталога сценариев

Цель: позволить пользователю просматривать небольшой каталог Git-упражнений и выбирать одно из них для открытия.

Сюда входит работа вроде:

- поставки summary-данных сценариев для первых MVP-упражнений
- публикации начального каталога из backend или локальных fixtures
- рендера списка каталога, фильтров и действий входа
- обработки состояний loading, empty и error при просмотре каталога

### Родительская задача 2. MVP контекста упражнения и workspace

Цель: позволить пользователю открыть упражнение и понять задачу, ситуацию в репозитории и ожидаемую цель до отправки ответа.

Сюда входит работа вроде:

- поставки payload деталей сценария и контекста репозитория
- рендера инструкций задачи, step-content и подсказок по репозиторию
- wiring exercise route и оболочки workspace
- поддержания понятного экрана упражнения ещё до добавления валидации

### Родительская задача 3. MVP отправки ответа и проверки корректности

Цель: позволить пользователю отправить ответ и немедленно получить результат проверки.

Сюда входит работа вроде:

- запуска exercise session и приёма пользовательских submissions
- валидации первых поддержанных answer types
- рендера answer input, pending states и correctness feedback
- возврата machine-readable outcome-модели, которую UI сможет понятно показать

### Родительская задача 4. MVP управляемого повтора и подсказок

Цель: помочь пользователю восстанавливаться после ошибок без потери контекста.

Сюда входит работа вроде:

- возврата объяснений для неправильных или частично правильных ответов
- сохранения контекста упражнения между повторными попытками
- показа более сильных подсказок после повторяющихся ошибок
- превращения цикла retry в обучающий, а не бинарный

### Родительская задача 5. MVP прогресса и рекомендаций следующего шага

Цель: сделать повторяемую практику видимой, чтобы пользователь видел свой прогресс и понимал, что делать дальше.

Сюда входит работа вроде:

- сохранения результатов попыток и состояния завершённости
- показа решённых и начатых упражнений
- суммарного представления прогресса пользователя во времени
- рекомендаций подходящих следующих упражнений
