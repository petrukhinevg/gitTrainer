# Рабочий workflow

## Назначение

- Этот файл про scope, декомпозицию и шаблоны issue.
- Ветки и PR: `docs/agent/GIT_WORKFLOW.md`.
- Статусы и доска: `docs/agent/BOARD_WORKFLOW.md`.

## Правила

- Одна задача = один законченный reviewable результат.
- Если результат нельзя описать одной короткой фразой, дели задачу.
- Используй roadmap как источник новых issue.
- Если крупного изменения нет в roadmap, сначала обнови roadmap.
- Не смешивай исследование и реализацию, если их можно разделить.

## Когда нужен `parent issue`

- несколько независимых user flows
- один change слишком велик для одного review
- работа естественно делится на независимые результаты

## Декомпозиция

- начинай с минимального набора независимых sub-issues
- дроби дальше только если это улучшает параллельность или reviewability
- дели по результатам, а не по framework-слоям
- не заводи sub-issue без самостоятельной ценности для review
- для этого продукта обычно разделяй content, validation и SPA UX

## Шаблон задачи

- язык: русский
- название: короткое, без `[эпик]`
- результат
- критерии готовности
- ограничения
- сторона и labels

## Шаблон parent issue

- язык: русский
- название: `[эпик] <текст>`
- цель
- `Sub-issues`
- граница
- порядок
